Accessibility Assessment

Summaries of 13 WCAG success criteria and a rationale for which apply to this project, linked to our Client & User Needs.

Overview

This page documents the results of the Accessibility Assessment activity: concise summaries of the applicable WCAG success criteria and our selection rationale based on previously identified Client and User Needs.

Scope & references used
  • WCAG Quick Reference (focus on 13 criteria from L04–L06).
  • Client Needs (Week 5) and User Needs (interviews with GenAI).

WCAG Success Criteria Summaries

Each item includes a plain-language aim and specific actions most relevant to our context. (Keep bullets short and actionable.)

Principle: Perceivable

1.1.1 Alternative text for non-text content

Aim: Ensure non-text content has text alternatives.

  • Provide concise, meaningful alt for informative images.
  • Use empty alt for decorative images.
  • Label form icons/controls programmatically.

1.3.2 Meaningful Sequence

Aim: Reading/navigation order matches visual order.

  • Use semantic HTML and logical DOM order.
  • Avoid layout that requires tabindex hacks for order.

1.4.1 Use More than Color

Aim: Don’t rely on color alone to convey meaning.

  • Add text, icons, or patterns in addition to color.
  • Ensure links are distinguishable beyond color (e.g., underline).

1.4.3 Contrast (Minimum)

Aim: Body text meets minimum contrast thresholds.

  • Text vs. background ≥ 4.5:1 (normal) or 3:1 (large text).
  • Test link text and iconography as well.

1.4.4 Resize Text

Aim: Content remains usable at 200% zoom.

  • Use responsive units; avoid fixed pixel heights.
  • Ensure no content overlap, truncation, or loss of function.

Principle: Operable

2.1.1 Keyboard

Aim: All functionality is keyboard accessible.

  • Ensure focusable controls; no keyboard traps.
  • Provide visible focus and logical tab order.

2.4.7 Focus Visible

Aim: Keyboard focus is clearly visible.

  • Custom focus styles with sufficient contrast and size.
  • Never remove outlines without providing a clear alternative.

Principle: Understandable

3.1.5 Reading Level

Aim: Provide supplemental support for complex content.

  • Use plain language; define jargon and acronyms.
  • Offer summaries or glossaries for advanced content.

3.2.3 Consistent Navigation

Aim: Navigation appears in a consistent order and location.

  • Keep menus, headers, and footers consistent across pages.
  • Avoid unexpected reordering between pages.

3.2.4 Consistent Identification

Aim: Identical components are identified consistently.

  • Label similar controls the same way across pages.
  • Keep icon meanings and behaviors consistent.

3.3.1 Error Identification

Aim: Clearly describe input errors and how to fix them.

  • Associate error messages with fields (aria-describedby).
  • Provide suggestions to correct format/value.

3.3.2 Labels or Instructions

Aim: Inputs have clear labels/instructions.

  • Use explicit <label for> or aria-label.
  • Provide format examples and helpful hints.

3.3.4 Error Prevention (Legal, Financial, Data)

Aim: Prevent/allow recovery from critical errors.

  • Confirmations for destructive/irreversible actions.
  • Review screens and easy undo where possible.

Relevance & Rationale

The following maps each selected Success Criteria to specific Client/User Needs and gives a brief rationale.

Mapping: Success Criteria → Client/User Needs → Rationale
Success Criteria Linked Need(s) Rationale (why this matters here)
1.1.1 Non-text Content Client: Showcase visuals; User: Screen reader users Alt text ensures imagery communicates meaning to non-visual users.
2.1.1 Keyboard User: Keyboard-only navigation All menus/forms remain operable without a mouse.
1.4.3 Contrast (Min) User: Readability in varied environments Meets minimum contrast for body text and links for legibility.
